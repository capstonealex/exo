3 ## Introduction
4 This document is a style guide to provide best practice formatting guidelines for ALEX Software Documentation using Doxygen. As such, the guide is a compilation of material from the [Doxygen Manual](http://www.doxygen.nl/manual/docblocks.html) and the [How to Write Doc Comments for the Javadoc Tool](https://www.oracle.com/technical-resources/articles/java/javadoc-tool.html) technical article. The latter document contains a Javadoc style guide which provides conventions for creating useful API documents - these conventions are included at the end of this document as an essential documentation reference.

42 ## Documenting A Function/Method {#docfunction}
43 ### Function Special Commands
44 
45 | Function Command | Decription | Doxygen Manual Link |
46 |----|----|----|
47 |`\param` | Used to describe a function parameter | [\param command](http://www.doxygen.nl/manual/commands.html#cmdparam) |
48 |`\return` | Used to describe the function return variable | [\return command](http://www.doxygen.nl/manual/commands.html#cmdreturn)
49 |`\sa` | Used to refer to associated functions, may be used to avoid unnecessary repitition | [\sa command](http://www.doxygen.nl/manual/commands.html#cmdsa)
50 
51 The following tips (sourced from the [Javadoc Style Guide](https://www.oracle.com/technical-resources/articles/java/javadoc-tool.html)) provide convention for using `\param` and `\return`:
52 * `\param` - Use the command `\param` followed by the name (not the data type) of the parameter, then the description. By convention:
53  * The first noun in the description is the data type of the parameter (may be preceded by an article "an", "a", "the"). Exception is the primitive type **int**
54  * Parameter names are lowercase
55 * `\return`
56  * **Omit** `\return`for methods that return void and for constructors
57  * **Include** `\return` for all other methods, even if its content is entirely redundant with the method description
58  * Having an explicit @return tag makes it easier for someone to find the return value quickly
59  * Whenever possible, supply return values for special cases (such as specifying the value returned when an out-of-bounds argument is supplied)
60

75 ## Documenting Enumerated Types {#docenum}
76 Source: [Doxygen Manual](http://www.doxygen.nl/manual/docblocks.html)
77 
78 ```c++
79 /**
80  * An enum type.
81  * The documentation block cannot be put after the enum!
82  */
83  enum EnumType
84  {
85  int Eval1, /**< enum value 1 */
86  int Eval2, /**< enum value 2 */
87  };
88 ```
89 [Doxygen Output - Documenting Enum Type](http://www.doxygen.nl/manual/examples/afterdoc/html/class_afterdoc___test.html)
90 
91 **Using In-line Comments to Display an Enum Table**
92 `/**< enum value 1 */` represents an inline comment. This comment is placed after the member it refers to, this association is indicated by `<` symbol in the comment description. Here, the placement of in-line comments associated the enum variable `Eval1` with its `value 1` and Doxygen creates an enum table from these associations.

107 ## Tables (Documenting Maps) - Markdown Support {#docmaps}
108 Doxygen supports the use of Markdown to create tables in the doc comments. These should be used to document mappings (key-value pairs) when an enum type is not used.
109 See the example below:
110 ```c++
111 /**
112  * First Header | Second Header
113  * ------------ | -------------
114  * Content Cell | Content Cell
115  * Content Cell | Content Cell
116  */
117  ```
118  For column alignment, one or two colons (`:`) can be used to select the type of alignment in the header seperator line.
119  See the example below:
120  ```c++
121  /**
122  * | Right | Center | Left |
123  * | ----: | :----: | :----|
124  * | value | value | value|
125  */
126 ```

153 ## Tips From The Javadoc Style Guide {#javastyleguidetips}
154 Source: [Javadoc Style Guide](https://www.oracle.com/technical-resources/articles/java/javadoc-tool.html)
155 The following tips from the Javadoc Style Guide are presented as either one-to-one translation from the guide (see source reference above) or a paraphrase of the original content.
156 1. **First Sentence** - The first sentence of each doc comment should be a summary sentence, containing a concise but complete description of the API item
157 2. **Comment Description** - Ideally, make it complete enough for conforming implementors. Realistically, include enough description so that someone reading the source code can write a substantial suite of conformance tests. Basically, the spec should be complete, including boundary conditions, parameter ranges and corner cases
158 3. **Use <code> style for keywords and names**
159 4. **Use in-line links economically**
160  * It is not necessary to add links for all API names in a doc comment. Add API links when:
161  * The user might actually want to click on it for more information
162  * It is the first occurrence of the API name in the doc comment
163 5. **Omit parentheses for the general form of methods and constructors** - only use when referring to the specific form (for example, when you wish to distinguish between two methods)
164 6. **OK to use phrases instead of complete sentences, in the interest of brevity**
165 7. **Use 3rd person (Descriptive) not 2nd person (prescriptive)** - Gets the label (preferred) over Get the label (avoid)
166 8. **Method descriptions begin with a verb phrase** - A method implements an operation, or performs an action, so it should begin with a verb phrase (for example, This method sends a position command to the `\code<Drive>` described by DriveID)
167 9. **Class/interface/field descriptions can omit the subject and simply state the object** - The aformentioned API often describe things rather than actions or behaviours, as such: *Drive class containing*... (preferred) over *This is a Drive class containing* (avoid)
168 10. **Use "this" instead of "the" when referring to an object created from the current class.** - Gets the toolkit for **this** component (preferred) over Gets the toolkit for **the** component (avoid)
169 11. **Add description beyond the API name** - The best API names are "self-documenting". Thus, it is not beneficial to simply repeat the API name in description. The ideal comment should reward the reader with additional information which is not immediately obvious from the API name.
170  **Avoid** - The description below says nothing beyond what you know from reading the method name. The words "set", "tool", "tip", and "text" are simply repeated in the sentence
171  ```java
172  /**
173  * Sets the tool tip text.
174  *
175  * @param text the text of the tool tip
176  */
177  public void setToolTipText(String text)
178  ```
179  **Preferred** - This description more completely defines what a tool tip is, in the larger context of registering and being displayed in response to the cursor.
180 
181  ```java
182  /**
183  * Registers the text to display in a tool tup. The text
184  * displays when the cursor lingers over the component.
185  *
186  * @param text the string to display. If the text is null,
187  the tool tip is turned off for this component.
188  */
189  public void setToolTipText(String text)
190  ```
191 12. **Tag Conventions** - **Note:** The equivalent Doxygen commands will be used in place of Javadoc commands (@ replaced by \\)
192  Order of Tags - Include tags in the following order
193  1. `\author` (classes and interfaces only, required)
194  2. `\version` (classes and interfaces only, required)
195  3. `\param` (methods and constructors only)
196  4. `\return` (methods only)
197  5. `\exception`(`\throws` is a synonym)
198  6. `\sa` or `\see\`
